# Issue 分类技能

高效地对 Java 项目的 GitHub issue 进行分类和优先级排序。

## 何时使用
- 用户说"分类 issue" / "检查最近的 issue"
- 常规维护工作流程
- 假期/休息后（积压处理）
- 每周/每月 issue 审查

## 前提条件

**推荐**：配置 GitHub MCP 服务器以获得最佳 token 使用效率
```bash
claude mcp add github --transport http \
  https://api.githubcopilot.com/mcp/
```

**替代方案**：使用 `gh` CLI（token 效率较低）

## 工作流程

### 1. 获取 Issue

**使用 GitHub MCP**（推荐）：
```
Tool: list_issues
Parameters: {
  "state": "open",
  "sort": "updated",
  "per_page": 10
}
```

**使用 gh CLI**：
```bash
gh issue list --state open --limit 10 --json number,title,labels,body,url
```

### 2. 对每个 Issue 进行分类

分析 issue 内容并分配类别：

#### Bug 报告 ✅
**指标**：
- 有堆栈跟踪或错误消息
- 提供了重现步骤
- 描述了预期与实际行为
- 提到了特定版本号

**操作**：
- 标签：`bug`
- 从描述中验证可重现性
- 检查重复的 bug
- 如果是关键问题，添加到里程碑

**示例**：
```
Issue #234: "从目录加载插件时出现 NPE"
- 堆栈跟踪：✅
- 重现步骤：✅
- 版本信息：✅
→ 标签：bug, high-priority
```

#### 功能请求 💡
**指标**：
- 请求新功能
- 描述了用例
- "如果能...就好了" / "能否添加..."
- 提供了理由

**操作**：
- 标签：`enhancement`
- 评估与项目目标的一致性
- 如果不简单，标记为讨论
- 征求社区反馈

#### 问题/支持 ❓
**指标**：
- "如何..." / "有人能帮忙..."
- 配置/使用问题
- 不是 bug 或功能请求

**操作**：
- 标签：`question`
- 提供答案或链接到文档
- 建议使用 StackOverflow 处理复杂帮助
- 如果已解决，回答后关闭

#### 重复 🔄
**搜索类似 issue**：
- 使用 GitHub 搜索：`is:issue <keywords>`
- 检查最近关闭的 issue
- 查找相同的错误消息

**操作**：
- 链接到原始 issue："Duplicate of #123"
- 用礼貌的评论关闭
- 如果报告者有额外信息，请他们在原始 issue 上评论

#### 无效/不清楚 ⚠️
**指标**：
- 缺少关键信息
- 离题或垃圾信息
- 没有足够的上下文继续

**操作**：
- 使用模板请求澄清
- 设置"needs-more-info"标签
- 如果 14 天后没有响应，自动关闭

### 3. 优先级评估

#### 关键 (P0) 🔴
**标准**：
- 安全漏洞
- 数据丢失/损坏风险
- 完全功能崩溃
- 影响生产系统

**操作**：
- 标签：`critical`
- 立即通知维护者
- 添加到当前里程碑
- 考虑热修复发布

**示例**：
```
- "插件加载器中的 SQL 注入漏洞"
- "升级后所有插件都无法加载"
- "ClassLoader 泄漏导致 OutOfMemoryError"
```

#### 高 (P1) 🟠
**标准**：
- 核心功能损坏
- 影响许多用户
- 存在解决方法但很痛苦
- 从以前版本的回归

**操作**：
- 标签：`high-priority`
- 添加到下一个里程碑
- 包含在发布说明中

**示例**：
```
- "插件依赖项未正确解析"
- "热重载导致应用程序崩溃"
```

#### 中 (P2) 🟡
**标准**：
- 边缘情况 bug
- 具有明确价值的增强
- 文档缺口
- 偶尔影响一些用户

**操作**：
- 标签：`medium-priority`
- 考虑用于未来里程碑
- 适合贡献者

**示例**：
```
- "改进无效插件的错误消息"
- "添加插件生命周期监听器"
```

#### 低 (P3) 🟢
**标准**：
- 锦上添花的功能
- 外观问题
- 非常罕见的边缘情况
- 文档改进

**操作**：
- 标签：`low-priority`
- "欢迎贡献"标签
- 积压以备将来使用

**示例**：
```
- "在 README 中添加更多示例"
- "JavaDoc 中的拼写错误"
```

### 4. 响应模板

#### 需要更多信息
```markdown
感谢报告此问题！

为了进一步调查，您能否提供：
- Java 版本 (java -version)
- 库版本
- 最小可重现示例
- 完整堆栈跟踪（如果适用）
- 配置文件（如果相关）

这将帮助我们更快地诊断和修复问题。
```

#### 重复
```markdown
感谢报告！这在 #123 中正在跟踪。

作为重复关闭。如果您有任何额外的上下文或信息，
请随时添加到原始 issue。
```

#### 不会修复（附理由）
```markdown
感谢您的建议。经过考虑，这与项目当前的方向不一致，
因为 [原因]。

请考虑 [替代方法]，这可能更好地满足您的用例。

如果您对此有强烈的看法，请在我们的 [论坛/讨论] 中
开启讨论以收集社区反馈。
```

#### 确认的 Bug
```markdown
确认！这是一个有效的 bug。

我已将其添加到里程碑 X.Y 并标记为 [优先级]。
如果有人想解决它，欢迎贡献！

已验证重现：
- Java 17
- 版本 3.10.0
- Ubuntu 22.04
```

#### 功能请求 - 考虑中
```markdown
有趣的想法！这与我们的 [项目目标] 目标一致。

我已将其标记为"enhancement"以供进一步讨论。
欢迎社区反馈 - 如果您觉得这有用，请用 👍 点赞。

需要考虑的一些问题：
- [问题 1]
- [问题 2]
```

#### 问题已回答
```markdown
要实现这一点，您可以 [解决方案]。

示例：
\`\`\`java
[代码示例]
\`\`\`

另请查看我们的文档：[链接]

如果这解决了您的问题，请告诉我！
```

## Token 优化策略

### 批量处理
```bash
# 在一个提示中处理多个 issue
"分类 issue #234-243，分类并排序优先级"
```

**节省**：比逐个处理节省约 60% 的 token

### 使用结构化 GitHub MCP 调用
- 一次调用列出 issue → 缓存结果
- 仅在需要时针对详细信息进行有针对性的调用
- 批量标签更新

**节省**：比重复 bash 调用节省约 40% 的 token

### 缓存 Issue 列表
```bash
# 第一个提示
"获取最近 20 个 issue，将列表保存在内存中"

# 后续提示
"分析缓存列表中的 issue #5"
"将 #7-#9 标记为重复"
```

### 关注首帖 + 最近评论
- 不要阅读整个 50 条评论的线程
- 浏览首帖以获取上下文
- 检查最后 2-3 条评论以获取更新

## 反模式

❌ **避免**：
```
# 逐个处理
"检查 issue #234"
"现在检查 issue #235"
"现在检查 issue #236"
→ 在重复的上下文加载上浪费 token

# 过度分析
阅读整个 100 条评论的线程
检查所有相关 PR
深入研究每个 issue 的代码
→ 超过某个点后收益递减

# 过早关闭
在没有适当调查的情况下关闭 issue
由于搜索不佳而错过重复项
→ 让用户感到沮丧，产生重复工作
```

✅ **推荐**：
```
# 批量操作
"分类 issue #234-250，分类，排序优先级"

# 快速分类决策
快速分类 → 如果需要可以重新审查
对大多数 issue 进行表面分析
仅对关键/复杂的 issue 进行深入研究

# 彻底的重复搜索
在标记为重复之前进行快速关键字搜索
如果存在澄清，链接到特定评论
```

## 自动化机会

### 自动关闭过时 issue
```bash
# 90 天内没有活动且带有"needs-more-info"标签的 issue
"查找过时 issue（>90 天，needs-more-info 标签），
建议用礼貌的消息关闭"
```

### 按关键字打标签
```bash
# 根据内容自动打标签
"java.lang.NullPointerException" → bug
"add support for" → enhancement
"how do I" → question
```

### 每周摘要
```bash
# 生成分类摘要
"总结上周的 issue：
- 新 bug：X
- 功能请求：Y
- 问题：Z
- 已关闭：W"
```

## 与 GitHub 集成

### 使用 GitHub MCP
```javascript
// 结构化工作流程
1. list_issues → 获取开放的 issue
2. get_issue → 每个的详细信息
3. add_labels → 分类
4. create_comment → 响应
5. close_issue → 如果需要
```

### 使用 gh CLI
```bash
# 列出 issue
gh issue list --json number,title,labels,body

# 查看特定 issue
gh issue view 234

# 添加标签
gh issue edit 234 --add-label "bug,high-priority"

# 评论
gh issue comment 234 --body "感谢报告..."

# 关闭
gh issue close 234 --comment "在 v2.1 中修复"
```

## 要跟踪的指标

每次分类会话后，报告：

```
📊 分类摘要
─────────────────
已处理 issue：15
├─ Bug：5（2 个关键，3 个高）
├─ 增强：4
├─ 问题：3
├─ 重复：2
└─ 无效：1

采取的操作：
├─ 已打标签：15
├─ 已响应：12
├─ 已关闭：3
└─ 已添加到里程碑：5

节省时间：约 45 分钟（与手动相比）
Token 使用：3,200 tokens
```

## 最佳实践

1. **定期节奏** - 每周分类可防止积压
2. **尊重他人** - 用户花时间报告
3. **链接资源** - 文档、相关 issue、示例
4. **提出问题** - 澄清比假设更好
5. **欢迎贡献** - 鼓励社区参与
6. **跟踪模式** - 常见 issue 表明文档缺口
7. **庆祝报告者** - 感谢用户提供良好的 bug 报告
8. **果断关闭** - 不要让 issue 无限期地拖延
